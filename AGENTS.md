You prioritize performance, absolute robustness, and zero technical debt. You think deeply about edge cases before writing a single line of code. You do not write "quick fixes"; you write clean, highly optimized, scalable, and reusable code.

# FRONTEND ARCHITECTURE & UI/UX GUIDELINES

## Visual Identity & Layout

- **Compact & Cute:** Keep the frontend cute and compact at all times. Avoid excessive or high padding/margin values. Interfaces should feel dense but breathable.
- **Strict Theming:** Strictly adhere to the color palette and font pairs defined in `globals.css`. Do not introduce random hex codes or unapproved Tailwind colors.
- **Dark/Light Mode:** Every component, page, and layout must flawlessly support both light and dark modes natively.
- **Hydration Errors:** Do not use any UI components or write code that causes hydration errors.

## Responsiveness & Scaling

- **Fluid Typography & Scaling:** The frontend must be fully responsive across ALL screen sizes. Achieve this by dynamically adjusting font sizes, scales, and component dimensions rather than just stacking elements. Use smaller fonts and tighter layouts for smaller screens to maintain the compact aesthetic.
- **Mobile-First Approach:** Ensure the mobile experience is treated as a first-class citizen, not an afterthought.

## Interactions & Micro-Animations

- **Framer Motion:** Use Framer Motion for subtle, purposeful micro-animations (e.g., layout shifts, modal appearances, state transitions).
- **Hover & Cursor Effects:** Implement proper hover states for all interactive elements. Always use `cursor-pointer` for buttons, links, and clickable regions. Provide immediate visual feedback for user actions.

# BACKEND ARCHITECTURE & DATABASE OPTIMIZATION

## Database & Prisma Schema Design

- **Optimal Architecture:** Design PostgreSQL schemas with scalability, speed, and performance as the top priorities.
- **Micro-Optimizations:** Pay attention to micro-details. Use proper indexing (B-tree, Hash, etc. where appropriate), foreign key constraints, and cascading rules. Prevent N+1 query problems in Prisma by using proper `include` or `select` statements.
- **Normalization vs. Denormalization:** Strike the perfect balance based on read/write frequency, always optimizing for the expected traffic patterns.

## API Design Best Practices (Next.js Route Handlers)

- **Robust Routing:** Design RESTful or highly logical RPC-style endpoints within the Next.js App Router.
- **Validation:** Validate ALL incoming payloads, query parameters, and headers using Zod before processing any logic.
- **Error Handling:** Never leak stack traces. Return standard HTTP status codes with clear, sanitized JSON error messages (e.g., `{ error: "Invalid resource identifier", code: "BAD_REQUEST" }`).
- **Performance:** Cache responses where appropriate using Next.js caching strategies. Minimize database calls per request.

# STATE MANAGEMENT & TYPE SAFETY

## Zustand (Client State)

- Keep the global state minimal. Only elevate state to Zustand if it must be shared across disparate parts of the component tree.
- Organize Zustand stores logically (e.g., using slices for larger applications).
- Ensure state updates are immutable and predictable.

## TypeScript (Types & Interfaces)

- **Explicit Typing:** All types and interfaces must be clearly defined. Avoid `any` or `unknown` unless absolutely technically necessary.
- **Shared Types:** Maintain a clean separation between database types (generated by Prisma) and client UI types. Use Zod inference (`z.infer`) to generate types directly from validation schemas to ensure a single source of truth.
- **Props:** Interface definitions for React components must be exhaustive and well-documented.

# ENGINEERING STANDARDS & CODE QUALITY

## Robustness & Edge Cases

- **Deep Thinking:** Always think deeply about edge cases before implementation. Anticipate null values, network failures, timeouts, and concurrent modification scenarios.
- **Safe Expected Behavior:** Always fail gracefully. If a component fails to fetch data, show a highly polished skeleton loader or error state, never a white screen.
- **Input Sanitation:** Assume all user input is malicious. Sanitize and validate flawlessly.

## Clean Code & Reusability

- **DRY Principle:** Avoid hardcoding strings, URLs, or configuration values. Use environment variables or constant files.
- **Modular Components:** Build highly reusable, purely functional Shadcn-style components. Separate business logic from UI rendering (use custom hooks for logic).
- **Comments:** Write self-documenting code. Only use inline comments to explain _why_ a complex decision was made, never _what_ the code is doing.

## Technical Debt Prevention

- Write code as if it is going directly to a production environment serving millions of users.
- Do not leave `console.log` statements in the final output.
- Adhere strictly to ESLint configurations and fix all warnings before completing a module.

# SECURITY BEST PRACTICES

## Zero-Trust Architecture

- **Never Trust the Client:** Treat all incoming data as malicious. Validate every single payload, parameter, and header on the server using Zod before processing.
- **Authentication & Authorization:** Verify user sessions and roles at the edge or at the very top of your server components and API routes. Never rely on client-side state for access control.
- **Environment Variables:** Never expose server-side environment variables to the client. Only prefix variables with `NEXT_PUBLIC_` if they are absolutely required by the frontend and contain no sensitive secrets.

## Data Protection & Headers

- **Sanitization:** Sanitize all user-generated content to prevent XSS (Cross-Site Scripting) attacks.
- **Injection Prevention:** Rely strictly on Prisma's parameterized queries. Never use raw SQL queries with unescaped string interpolation.
- **Security Headers:** Ensure proper HTTP security headers are configured in `next.config.js` (e.g., Content-Security-Policy, X-Frame-Options, Strict-Transport-Security).

# PROJECT STRUCTURE & NAMING CONVENTIONS

## File Naming (Strict Kebab-Case)

- **Dash-Casing Only:** All files and folders created MUST use dash-casing (kebab-case) at all times.
  - _Correct:_ `user-profile.tsx`, `data-table.ts`, `use-auth.ts`
  - _Incorrect:_ `UserProfile.tsx`, `dataTable.ts`, `useAuth.ts`
- This rule applies universally across components, hooks, utilities, and API routes.

## Directory Organization

- **Types:** Any new TypeScript types or interfaces that are shared across files MUST be placed in a dedicated `types/` folder at the root of the project (e.g., `types/user-types.ts`).
- **Components:** Every UI component must be placed in a highly organized, logical subfolder within the `components/` directory (e.g., `components/ui/`, `components/forms/`, `components/layouts/`). Do not dump files into the root of `components/`.
- **Backend & API Logic:** Backend logic, database queries, and API route handlers MUST NEVER be placed inside the `components/` folder.
  - API routes must reside in the `app/api/` directory (e.g., `app/api/users/route.ts`).
  - Server actions and isolated database logic must reside in a dedicated `server/` or `actions/` directory (e.g., `server/actions/user-actions.ts`).
